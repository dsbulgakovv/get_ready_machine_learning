# Production Handbook

Этот модуль закрывает инженерные темы вокруг сервисов: контейнеризация, монолит vs микросервисы, асинхронность, параллелизм и типичные практические сценарии вроде массовой выкачки изображений.

## Контейнеризация

### Для чего нужна контейнеризация сервиса?

Чтобы воспроизводимо упаковать код, зависимости и окружение. Это уменьшает разницу между локальной машиной, CI и продом.

### Какие задачи решает Docker при разработке?

- единое окружение;
- удобный запуск зависимостей;
- воспроизводимость;
- изоляция;
- упрощение деплоя.

## Архитектура сервисов

### Какие преимущества и недостатки у микросервисной архитектуры по сравнению с монолитом?

Плюсы микросервисов:

- независимый деплой;
- масштабирование отдельных частей;
- технологическая гибкость;
- более явные границы ответственности.

Минусы:

- сложнее наблюдаемость;
- сложнее отладка;
- выше инфраструктурные затраты;
- сетевые ошибки и распределённые транзакции.

### Основные отличия монолита и микросервиса? Преимущества микросервисов?

Монолит проще на старте и в отладке. Микросервисы выигрывают, когда система уже большая, команд несколько и модули действительно живут независимо.

### Что по поводу отказоустойчивости в этих архитектурах?

Микросервисы могут локализовать отдельные падения, но увеличивают число точек отказа. Монолит проще, но иногда один сбой бьёт сразу по всему сервису. Отказоустойчивость определяется не только типом архитектуры, но и качеством observability, retries, timeouts, circuit breakers и деплоя.

### Какие ты знаешь способы взаимодействия микросервисов между друг другом?

- HTTP/REST;
- gRPC;
- message brokers;
- event bus;
- shared storage, хотя это часто спорный вариант.

## Async, threads, processes

### Как работает асинхронность в Python? Что такое горутина? Что такое event loop? Что такое await?

В Python корректнее говорить `корутина`, а не "горутина". Корутина — `async def` функция, event loop управляет переключением задач, `await` отдаёт управление в точке ожидания неблокирующей операции.

### Для какого типа задач эффективно использовать асинхронность?

Для I/O-bound: сеть, БД, файловые операции, внешние API.

### Чем отличается мультитрединг от мультипроцессинга? В чем основное отличие треда и процесса?

- потоки делят память процесса;
- процессы изолированы;
- threads дешевле, processes дают реальный CPU parallelism.

### Зачем вообще нужен мультитрединг, если есть GIL?

Потому что GIL ограничивает CPU-bound Python bytecode, но для I/O-bound задач потоки всё равно полезны: пока один поток ждёт сеть, другой может работать.

### Почему в задаче с выкачкой картинок даст буст мультитрединг?

Потому что задача упирается не в CPU, а в ожидание сети и диска. Потоки позволяют перекрыть время ожидания.

### Что значит распараллелить несколько инпутов?

Это обрабатывать несколько независимых запросов/элементов параллельно, а не последовательно, чтобы улучшить throughput и/или latency.

## Практический сценарий

### Как бы ты выкачивал датасет картинок, если есть сервис с ручкой, которая высылает картинки?

Я бы сделал так:

1. очередь URL/ID изображений;
2. ограниченный пул воркеров;
3. таймауты, retries, exponential backoff;
4. идемпотентная запись;
5. отдельный лог неудачных скачиваний;
6. checksum/валидация;
7. батчевую постобработку после скачивания.

Для I/O можно использовать async или threads, для CPU-постобработки — processes.

## Базы данных в проде

### Какие бывают типы БД? В чем разница SQL/noSQL?

См. модуль БД. На продовом интервью хороший ответ: SQL — когда важны схема, целостность и сложные запросы; NoSQL — когда важны гибкость модели, горизонтальное масштабирование или конкретный тип доступа.

## Что важно запомнить по модулю

- Контейнеризация решает воспроизводимость, а не магически "ускоряет" сервис.
- Монолит vs микросервисы — это инженерный компромисс, а не религия.
- Для I/O-bound задач threads/async реально помогают даже в CPython.
- В сильном ответе на продовую тему почти всегда должны звучать `retry`, `timeout`, `idempotency`, `monitoring`.
