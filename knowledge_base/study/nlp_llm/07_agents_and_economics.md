## Учебник

### AI Agents

Агент — это система, где LLM не просто выдаёт текст, а умеет планировать, вызывать инструменты и использовать наблюдения из внешнего мира для достижения цели.

Типичный цикл:

1. получить цель;
2. спланировать шаг;
3. выбрать tool или действие;
4. получить observation;
5. обновить состояние;
6. повторять до завершения.

Это и есть базовый reasoning loop: thought -> action -> observation. Он лежит в основе паттернов Act/ReAct.

Если отвечать сильно, лучше не просто перечислить ярлыки, а разложить reasoning patterns по смыслу:

- `reactive / step-by-step loop`: агент смотрит на текущее состояние и принимает следующее действие без длинного глобального плана;
- `plan-and-execute`: сначала строит план, потом исполняет его по шагам;
- `reflection / self-critique`: после ответа или шага проверяет себя и корректирует стратегию;
- `decomposition`: разбивает большую задачу на подзадачи и решает их отдельно;
- `search-based reasoning`: перебирает несколько веток, гипотез или кандидатов и потом выбирает лучшую;
- `tool-grounded reasoning`: часть рассуждения выносит во внешний tool, например поиск, калькулятор, код или БД.

Когда какой паттерн полезен:

- reactive loop хорош для коротких tool-use сценариев;
- plan-and-execute полезен для длинных многошаговых задач;
- reflection помогает на сложных задачах с высоким риском ошибки;
- decomposition полезен, когда задачу можно естественно распилить на независимые части;
- search/tree-style reasoning нужен, когда одна цепочка рассуждений может рано уйти не туда;
- tool-grounded reasoning нужен, когда модель не должна полагаться только на внутреннюю память.

На собесе хороший короткий ответ: у агента есть reactive reasoning, planning, reflection, decomposition и tool-grounded reasoning; выбор паттерна зависит от длины задачи, цены ошибки и необходимости обращаться к внешнему миру.

### Экономическая целесообразность

Это задача не "про цену за токен", а про полный `TCO` и соответствие продуктовым требованиям.

Сравнивать нужно по нескольким осям:

- `качество`: насколько API-модель или self-hosted модель решает задачу;
- `latency и SLA`: укладывается ли решение в нужное время ответа и требования по доступности;
- `объём трафика`: средний и пиковый QPS / TPM;
- `стоимость`: токены у API против GPU, orchestration, storage, monitoring и on-call у self-hosted;
- `privacy / compliance`: можно ли вообще отправлять данные наружу;
- `операционная сложность`: есть ли команда, которая умеет держать inference в проде;
- `гибкость`: нужен ли контроль над моделью, fine-tuning, custom routing, KV-cache, batching;
- `vendor risk`: lock-in, quota limits, изменение цен, недоступность региона.

Практически я бы оценивал так:

1. замерить качество и latency обоих вариантов на реальных запросах;
2. посчитать реальный объём токенов и пики нагрузки;
3. посчитать месячный `Cost_API`;
4. посчитать месячный `Cost_self` с infra + ops + engineering;
5. сравнить решения не только по цене, но и по privacy, SLA и velocity команды.

Хорошая собеседовательная формулировка: API почти всегда выигрывает на старте и при низкой/средней нагрузке, а self-hosting начинает выигрывать, когда трафик стабилен, требования понятны, а команда готова нести инфраструктурную сложность.

Примерно:

$$
\text{Cost}_{API} = N_{\text{in}} \cdot p_{\text{in}} + N_{\text{out}} \cdot p_{\text{out}}
$$

где \(N_{\text{in}}, N_{\text{out}}\) — число входных и выходных токенов, \(p\) — цена за токен.

Нужно учитывать:

- GPU/CPU;
- storage;
- orchestration;
- разработку и поддержку;
- MLOps/monitoring;
- время инженеров;
- простой и резервирование.

Грубо:

$$
\text{Cost}_{self} = \text{infra} + \text{ops} + \text{engineering} + \text{downtime risk}
$$

Если раскрыть это по-взрослому, то в `infra` обычно входят:

- аренда или амортизация GPU;
- CPU/RAM для preprocessing и serving;
- storage для моделей, логов и артефактов;
- сеть и egress;
- резервные мощности под пик и failover.

В `ops` и `engineering` входят:

- деплой и обновление моделей;
- мониторинг, алертинг, on-call;
- autoscaling, batching, KV-cache, quantization;
- debugging latency и OOM;
- retraining / evaluation / rollback pipeline.

То есть self-hosting часто выглядит дешёвым только если считать одну GPU-карту и не считать поддержку.

Базовый вариант:

1. один supervisor/router;
2. несколько специализированных worker-агентов;
3. shared memory или state store;
4. правила handoff между агентами;
5. мониторинг и fallback.

<details>
<summary>Что такое Shared Memory?</summary>

Это общее состояние, доступное нескольким агентам или шагам. Там могут храниться план, промежуточные результаты, history, контекст пользователя, список уже вызванных tools, документы и артефакты.

Главная инженерная мысль: shared memory нужна не ради "умности", а чтобы агенты не теряли состояние и не дублировали работу.

</details>

- storage для моделей, логов и артефактов;
- сеть и egress;
- резервные мощности под пик и failover.

В `ops` и `engineering` входят:

- деплой и обновление моделей;
- мониторинг, алертинг, on-call;
- autoscaling, batching, KV-cache, quantization;
- debugging latency и OOM;
- retraining / evaluation / rollback pipeline.

То есть self-hosting часто выглядит дешёвым только если считать одну GPU-карту и не считать поддержку.

</details>

<details>
<summary>В какой момент выгоднее перейти на собственный сервер?</summary>

Переход становится выгодным, когда одновременно выполняются несколько условий:

- трафик достаточно большой и стабильный, чтобы загрузка GPU была высокой, а не рваной;
- `Cost_API` стабильно выше `Cost_self` с учётом полной эксплуатации;
- качество своей модели не хуже настолько, чтобы экономия не съедалась продуктовой деградацией;
- есть требования по privacy, locality данных или latency, которые плохо закрываются внешним API;
- команда умеет поддерживать inference stack и дежурства.

Хорошо звучит break-even логика:

$$
\text{если } \text{Cost}_{API}(month) > \text{Cost}_{self}(month)
$$

не разово, а устойчиво, при сопоставимом качестве и SLA, то self-hosting становится кандидатом.

Но на интервью важно добавить, что break-even по деньгам — не единственный критерий. Даже если self-hosting чуть дешевле, API может оставаться выгоднее, если:

- команда маленькая;
- продукт быстро меняется;
- нужно часто пробовать новые модели;
- операционный риск простоя слишком дорог.

То есть переход оправдан, когда economics, качество, latency и зрелость команды одновременно сходятся в пользу своей инфраструктуры.

</details>
