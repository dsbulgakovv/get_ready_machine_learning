## Учебник

### База языка

Основные:

- `int`, `float`, `bool`, `complex`;
- `str`, `bytes`;
- `list`, `tuple`, `set`, `dict`;
- `NoneType`;
- пользовательские классы.

Если отвечать шире, в Python часто отдельно вспоминают `range`, `frozenset`, `deque`, `memoryview`, итераторы и генераторы как важные встроенные абстракции.

Python динамически типизирован: тип присваивается объекту, а не имени переменной. Переменная может быть в любой момент пере-связана с объектом другого типа. При желании можно добавлять type hints для читаемости, IDE и статического анализа:

```python
name: str = "Alice"
scores: list[int] = [1, 2, 3]
```

- `mutable`: list, dict, set, большинство пользовательских объектов;
- `immutable`: int, float, str, tuple, frozenset.

Различие важно из-за ссылочной модели: изменение mutable-объекта видно через все ссылки на него. Ещё это влияет на хешируемость, возможность использовать объект как ключ словаря и на поведение при передаче в функции.

### Функции, декораторы, контекстные менеджеры

Метод — это функция, связанная с объектом или классом. При вызове экземплярного метода первым аргументом неявно передаётся `self`, а у `@classmethod` — `cls`. Это часть механизма descriptors, из-за которого доступ к атрибуту класса и экземпляра ведёт себя по-разному.

Это анонимная функция из одного выражения. Удобна для коротких inline-операций, особенно вместе с `sorted(key=...)`, `map` и `filter`. Ограничения:

- нельзя писать многострочный блок;
- хуже читается, если логика сложнее одной простой трансформации;
- у `def` есть имя, docstring и обычно лучше отладка.

Обе концепции нужны для оборачивания поведения:

- декоратор оборачивает функцию/метод;
- контекстный менеджер оборачивает блок кода.

### ООП и data model

Это специальные методы Python data model: `__new__`, `__init__`, `__str__`, `__repr__`, `__len__`, `__iter__`, `__next__`, `__enter__`, `__exit__` и т.д. Они позволяют объекту "вписаться" в язык и управляют тем, как объект участвует в `print`, `len`, `for`, `with`, `[]`, `+` и других операциях.

ООП — это способ моделировать программу через объекты, которые объединяют состояние и поведение. Классические принципы такие:

- `инкапсуляция`: объект скрывает внутренние детали и даёт наружу понятный интерфейс;
- `наследование`: можно переиспользовать и расширять поведение базового класса;
- `полиморфизм`: код может работать с объектами разных типов через общий интерфейс;
- `абстракция`: наружу выносится существенное, а детали реализации прячутся.

Для Python на собесе полезно добавить две оговорки:

- Python часто опирается не на жёсткие интерфейсы, а на `duck typing`;
- на практике часто предпочитают `composition over inheritance`, потому что глубокие иерархии классов быстро усложняют код.

Ещё одна важная Python-специфика: "приватность" здесь в основном договорная, через `_name` и name mangling, а не жёстко enforced как в некоторых других языках.

Класс — шаблон для создания объектов. Он объединяет данные и поведение: атрибуты экземпляра, методы, свойства и иногда class-level state. При создании объекта обычно сначала вызывается `__new__`, потом `__init__`.

```python
class User:
    def __init__(self, name: str):
        self.name = name
```

### Итераторы и генераторы

Итератор — объект с методами `__iter__` и `__next__`. Он хранит собственное состояние обхода и выдаёт элементы по одному. Когда элементы заканчиваются, `__next__` обязан бросить `StopIteration`.

```python
class Counter:
    def __init__(self, n):
        self.i = 0
        self.n = n

    def __iter__(self):
        return self

    def __next__(self):
        if self.i >= self.n:
            raise StopIteration
        value = self.i
        self.i += 1
        return value
```

Генератор — это функция с `yield`. Он сам реализует протокол итератора и позволяет лениво выдавать значения без ручного класса. Плюсы: меньше boilerplate, ленивое потребление, удобен для пайплайнов. Минус: иногда сложнее переиспользовать и дебажить, чем явный класс.

Да.

И наоборот неверно: не каждый итератор является генератором, потому что итератор можно реализовать вручную через класс.

### Асинхронность, потоки, процессы

Корутина — функция, объявленная через `async def`, которую исполняет event loop. Вызов `async def` не выполняет код сразу, а создаёт awaitable-объект, который надо либо `await`-ить, либо запускать через `asyncio.run`/task.

`event loop` — диспетчер задач, который переключает выполнение между корутинами. `await` говорит: "здесь можно подождать неблокирующую операцию и пока дать поработать другим задачам". Важный caveat: async эффективен только если ключевые I/O-вызовы тоже неблокирующие.

Для I/O-bound задач:

- HTTP-запросы;
- сокеты;
- БД;
- чтение сетевых сервисов.

Для CPU-bound задач async не решает проблему производительности сам по себе. Если внутри корутины есть длинный синхронный CPU-блок, он всё равно заблокирует event loop.

### Память и GC

В CPython есть reference counting плюс garbage collector для циклических ссылок. Объект удаляется, когда счётчик ссылок падает до нуля, а циклы добирает GC. Практический caveat: финальный `__del__` и циклы ссылок могут делать момент освобождения неочевидным.

Сразу при `refcount == 0`, либо позже проходом cyclic GC, если есть циклические ссылки.

На интервью полезно добавить: объекты с циклическими ссылками и `__del__` исторически tricky для GC, поэтому лучше не строить архитектуру на неявном моменте удаления.

### Практические инженерные вопросы

Нужно построить устойчивый pipeline:

1. читать ссылки батчами;
2. ограничивать concurrency;
3. делать retries и таймауты;
4. логировать неудачи отдельно;
5. сохранять метаданные и статус скачивания;
6. отделить скачивание от постобработки через очередь.

И обязательно:

- лимитировать concurrency, чтобы не убить внешний сервис;
- делать retry только для transient ошибок;
- хранить checkpoint, чтобы можно было продолжить после падения;
- писать checksum/валидацию скачанного файла;
- собирать метрики success rate и latency.

Для сетевого I/O хорошо подходят:

- `asyncio` + `aiohttp`;
- thread pool;
- очередь задач.

Если нужно ещё и CPU-предобработка, скачивание можно делать async/threaded, а тяжёлую обработку вынести в процессы.

Для очень большой нагрузки часто комбинируют: async или threads на I/O, processes на CPU, и отдельную очередь между этапами.

Я бы строил end-to-end pipeline примерно так:

1. пройтись по API c авторизацией, пагинацией и батчевой выгрузкой ID/URL;
2. положить задачи скачивания в очередь;
3. ограничить concurrency и соблюсти rate limit сервиса;
4. для сетевого I/O использовать async или thread pool;
5. сохранять файлы в предсказуемую структуру каталогов и отдельно писать метаданные;
6. валидировать ответ: статус, MIME type, размер, checksum, возможность открыть изображение;
7. CPU-предобработку, например resize или конвертацию, вынести в отдельный этап или process pool;
8. вести checkpointing, чтобы после сбоя продолжить с последнего обработанного батча;
9. логировать permanent failures отдельно от transient failures;
10. собрать мониторинг: throughput, error rate, retry rate, latency, corrupted files.

Ключевые инженерные детали, которые хорошо звучат на собесе:

- `retry` с exponential backoff только для временных ошибок;
- идемпотентная запись, чтобы перезапуск не портил результат;
- deduplication, если один и тот же объект может прийти несколько раз;
- разделение I/O и CPU этапов;
- хранение сырого датасета и обработанной версии раздельно.

### Небольшие coding patterns

```python
def matvec(matrix, vector):
    return [sum(a * b for a, b in zip(row, vector)) for row in matrix]
```

Сложность: \(O(nm)\), где \(n\) — число строк, \(m\) — длина вектора.

```python
def merge_sorted(a, b):
    i = j = 0
    out = []
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            out.append(a[i])
            i += 1
        else:
            out.append(b[j])
            j += 1
    out.extend(a[i:])
    out.extend(b[j:])
    return out
```

Сложность: \(O(n + m)\) по времени и \(O(n + m)\) по памяти на результат.

Идея:

1. найти `min`, `max`;
2. проверить, что \((max-min)\) делится на \(n-1\);
3. вычислить шаг;
4. проверить, что все ожидаемые элементы есть.

Это делается за \(O(n)\) по времени и обычно за \(O(n)\) по памяти через `set`, если не оптимизируем память любой ценой.

<details>
<summary>Как работает словарь (хеш-таблица)? Что такое хеш-функция и коллизии? Как с ними бороться?</summary>

Коллизия — ситуация, когда разные ключи попадают в один слот или цепочку пробинга. В CPython используется open addressing, а порядок вставки сохраняется как языковая гарантия начиная с Python 3.7.

</details>

<summary>Как устроен декоратор внутри? Что он принимает на вход и что возвращает?</summary>

```python
def deco(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper
```

```python
from functools import wraps

def log_calls(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        print(f"calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper
```

</details>

<details>
<summary>Как работают методы __enter__ и __exit__?</summary>

- `__enter__` вызывается при входе в `with`;
- `__exit__` — при выходе, даже если было исключение.

`__exit__` получает информацию об исключении и может его подавить, вернув `True`, но на практике исключения обычно не скрывают, а используют `with` именно для безопасного освобождения ресурса.

</details>

<details>
<summary>Реализуйте простой контекстный менеджер.</summary>

```python
class Timer:
    def __enter__(self):
        import time
        self.start = time.time()
        return self

    def __exit__(self, exc_type, exc, tb):
        import time
        self.elapsed = time.time() - self.start
        print(self.elapsed)
```

Альтернатива в реальном коде — `contextlib.contextmanager`, если нужен более лаконичный генераторный стиль.

</details>

<details>
<summary>В чём отличие функции от метода?</summary>

Метод — это функция, связанная с объектом или классом. При вызове экземплярного метода первым аргументом неявно передаётся `self`, а у `@classmethod` — `cls`. Это часть механизма descriptors, из-за которого доступ к атрибуту класса и экземпляра ведёт себя по-разному.

</details>

<details>
<summary>Что такое lambda-функция? Чем она отличается от обычной функции и когда создаётся?</summary>

Это анонимная функция из одного выражения. Удобна для коротких inline-операций, особенно вместе с `sorted(key=...)`, `map` и `filter`. Ограничения:

- нельзя писать многострочный блок;
- хуже читается, если логика сложнее одной простой трансформации;
- у `def` есть имя, docstring и обычно лучше отладка.

</details>

<details>
<summary>Что такое магические методы? Почему они нужны и какие вы знаете? (например, __str__, __repr__)</summary>

Это специальные методы Python data model: `__new__`, `__init__`, `__str__`, `__repr__`, `__len__`, `__iter__`, `__next__`, `__enter__`, `__exit__` и т.д. Они позволяют объекту "вписаться" в язык и управляют тем, как объект участвует в `print`, `len`, `for`, `with`, `[]`, `+` и других операциях.

</details>

<details>
<summary>Что такое итератор? Как создать класс-итератор? (методы __iter__ и __next__)</summary>

Итератор — объект с методами `__iter__` и `__next__`. Он хранит собственное состояние обхода и выдаёт элементы по одному. Когда элементы заканчиваются, `__next__` обязан бросить `StopIteration`.

```python
class Counter:
    def __init__(self, n):
        self.i = 0
        self.n = n

    def __iter__(self):
        return self

    def __next__(self):
        if self.i >= self.n:
            raise StopIteration
        value = self.i
        self.i += 1
        return value
```

</details>

<details>
<summary>Что такое генератор? Чем он отличается от итератора? (функция с yield vs класс)</summary>

Генератор — это функция с `yield`. Он сам реализует протокол итератора и позволяет лениво выдавать значения без ручного класса. Плюсы: меньше boilerplate, ленивое потребление, удобен для пайплайнов. Минус: иногда сложнее переиспользовать и дебажить, чем явный класс.

</details>

<details>
<summary>Верно ли, что любой генератор является итератором?</summary>

Да.

И наоборот неверно: не каждый итератор является генератором, потому что итератор можно реализовать вручную через класс.

</details>

<details>
<summary>Реализуйте итератор, который возводит значения в квадрат.</summary>

```python
class SquareIter:
    def __init__(self, values):
        self.values = values
        self.idx = 0

    def __iter__(self):
        return self

    def __next__(self):
        if self.idx >= len(self.values):
            raise StopIteration
        value = self.values[self.idx] ** 2
        self.idx += 1
        return value
```

Сложность: \(O(n)\) по времени, \(O(1)\) по дополнительной памяти.

</details>

<details>
<summary>Реализуйте итератор для чисел Фибоначчи.</summary>

```python
class Fib:
    def __init__(self, n):
        self.n = n
        self.i = 0
        self.a, self.b = 0, 1

    def __iter__(self):
        return self

    def __next__(self):
        if self.i >= self.n:
            raise StopIteration
        self.i += 1
        val = self.a
        self.a, self.b = self.b, self.a + self.b
        return val
```

Если на собесе спросят, как сделать Pythonic-версию, можно сразу сказать: для такого случая чаще пишут генератор через `yield`, а не класс.

</details>

<details>
<summary>Что такое ООП и его основные принципы?</summary>

ООП — это способ моделировать программу через объекты, которые объединяют состояние и поведение. Классические принципы такие:

- `инкапсуляция`: объект скрывает внутренние детали и даёт наружу понятный интерфейс;
- `наследование`: можно переиспользовать и расширять поведение базового класса;
- `полиморфизм`: код может работать с объектами разных типов через общий интерфейс;
- `абстракция`: наружу выносится существенное, а детали реализации прячутся.

Для Python на собесе полезно добавить две оговорки:

- Python часто опирается не на жёсткие интерфейсы, а на `duck typing`;
- на практике часто предпочитают `composition over inheritance`, потому что глубокие иерархии классов быстро усложняют код.

Ещё одна важная Python-специфика: "приватность" здесь в основном договорная, через `_name` и name mangling, а не жёстко enforced как в некоторых других языках.

</details>

<details>
<summary>Что такое класс? Как объявить и использовать?</summary>

Класс — шаблон для создания объектов. Он объединяет данные и поведение: атрибуты экземпляра, методы, свойства и иногда class-level state. При создании объекта обычно сначала вызывается `__new__`, потом `__init__`.

```python
class User:
    def __init__(self, name: str):
        self.name = name
```

</details>

<details>
<summary>Что такое super? Когда и зачем его применять?</summary>

Позволяет обратиться к следующей реализации в MRO, а не просто к "родителю" в лоб. Это особенно важно в множественном наследовании, где `super()` помогает корректно строить cooperative inheritance и не зашивать жёсткую связь на конкретный базовый класс.

</details>

<details>
<summary>В чём разница между @classmethod и @staticmethod?</summary>

- `@classmethod` получает `cls` и работает с классом;
- `@staticmethod` не получает ни `self`, ни `cls`, это просто функция внутри namespace класса.

Обычно `classmethod` используют для альтернативных конструкторов и логики, завязанной на состояние класса, а `staticmethod` — для утилит, которые логически относятся к классу, но не требуют доступа ни к экземпляру, ни к самому классу.

</details>

<details>
<summary>Что такое @dataclass? Зачем нужен?</summary>

Это удобный способ объявлять классы-структуры данных без лишнего boilerplate. Автоматически генерирует `__init__`, `__repr__`, сравнение и другое.

Полезные опции, которые часто всплывают на интервью:

- `frozen=True` для неизменяемых объектов;
- `order=True` для ordering-методов;
- `slots=True` для экономии памяти и более строгой модели атрибутов.

</details>

<details>
<summary>Как проверить, есть ли у объекта атрибут? (getattr, setattr, delattr)</summary>

- `hasattr(obj, "x")`
- `getattr(obj, "x", default)`
- `setattr(obj, "x", value)`
- `delattr(obj, "x")`

`getattr` с `default` часто безопаснее, чем `hasattr` + отдельный `getattr`, потому что это одна попытка доступа вместо двух.

</details>

<details>
<summary>Чем отличаются методы __str__ и __repr__?</summary>

- `__str__`: человекочитаемое представление;
- `__repr__`: более техническое представление, желательно пригодное для отладки.

Хорошее правило: `__repr__` должен быть максимально однозначным и по возможности близким к восстановлению объекта.

</details>

<details>
<summary>Как работает асинхронность в Python? Что такое корутина?</summary>

Корутина — функция, объявленная через `async def`, которую исполняет event loop. Вызов `async def` не выполняет код сразу, а создаёт awaitable-объект, который надо либо `await`-ить, либо запускать через `asyncio.run`/task.

</details>

<details>
<summary>Что такое event loop и оператор await?</summary>

`event loop` — диспетчер задач, который переключает выполнение между корутинами. `await` говорит: "здесь можно подождать неблокирующую операцию и пока дать поработать другим задачам". Важный caveat: async эффективен только если ключевые I/O-вызовы тоже неблокирующие.

</details>

<details>
<summary>Для каких задач эффективна асинхронность?</summary>

Для I/O-bound задач:

- HTTP-запросы;
- сокеты;
- БД;
- чтение сетевых сервисов.

Для CPU-bound задач async не решает проблему производительности сам по себе. Если внутри корутины есть длинный синхронный CPU-блок, он всё равно заблокирует event loop.

</details>

<details>
<summary>Что такое GIL и почему Python называют однопоточным?</summary>

GIL — Global Interpreter Lock, который позволяет только одному потоку выполнять Python bytecode одновременно в процессе CPython. Он упрощает внутреннюю работу интерпретатора, делает reference counting проще и снижает стоимость синхронизации внутри VM.

На собесе важно сказать, что GIL не запрещает конкурентность вообще: I/O-потоки, C-расширения и multiprocessing по-прежнему работают.

</details>

<details>
<summary>Чем отличается многопоточность от мультипроцессинга?</summary>

Короткий сильный ответ такой:

- `multithreading` — несколько потоков внутри одного процесса с общей памятью;
- `multiprocessing` — несколько отдельных процессов с изолированной памятью.

Практические различия:

- потоки дешевле по памяти и переключению;
- процессы дороже запускать и синхронизировать;
- потоки проще делят общие структуры данных, но это повышает риск гонок;
- процессы требуют IPC и сериализации, зато лучше изолируют сбои;
- в CPython потоки для `CPU-bound` задач ограничены GIL, а процессы позволяют использовать несколько ядер реально параллельно.

Обычно threads берут для I/O-bound задач, а processes — для CPU-bound задач и тяжёлых вычислений.

</details>

<details>
<summary>В чём различие между потоком и процессом?</summary>

Поток — это единица выполнения внутри процесса. Несколько потоков:

- разделяют одно адресное пространство;
- видят одни и те же объекты в памяти;
- разделяют файловые дескрипторы и часть process-level ресурсов.

Процесс — это отдельная единица исполнения со своей памятью и своим состоянием. Из-за этого:

- процесс лучше изолирован;
- падение одного процесса реже напрямую валит другой;
- обмен данными между процессами дороже;
- но именно процессы дают нормальный CPU parallelism в CPython.

Если хотят короткую инженерную формулировку: поток дешевле, но опаснее с точки зрения shared state; процесс тяжелее, но безопаснее и лучше для CPU-bound workloads.

</details>

<details>
<summary>Какие проблемы возникают при многопоточности?</summary>

- race conditions;
- deadlocks;
- starvation;
- сложности отладки;
- состояние, разделяемое между потоками.

Если хотят глубже, можно упомянуть ещё livelock, priority inversion и необходимость аккуратно выбирать mutex/lock/queue design.

</details>

<details>
<summary>Как организовать обмен данными между процессами?</summary>

- `multiprocessing.Queue`;
- `Pipe`;
- shared memory;
- сокеты;
- внешняя БД/брокер сообщений.

Чем больше и тяжелее объект, тем сильнее чувствуется стоимость сериализации и копирования между процессами.

</details>

<details>
<summary>Что такое ссылочная модель в Python?</summary>

Имя переменной хранит ссылку на объект. Операция присваивания не копирует объект, а привязывает имя к существующему объекту. Поэтому `a = b` не создаёт новый объект, а `id(a)` и `id(b)` могут совпадать, если это один и тот же объект.

</details>

<details>
<summary>В чём разница между copy и deepcopy?</summary>

- `copy.copy`: поверхностная копия, вложенные объекты общие;
- `copy.deepcopy`: рекурсивно копирует и вложенные объекты.

Практический caveat: `deepcopy` может быть дорогим по времени и памяти, а для некоторых объектов требует аккуратной настройки `__deepcopy__`. На собесе хорошо звучит мысль, что архитектурно лучше избегать лишних копий, чем лечить всё `deepcopy`.

</details>

<details>
<summary>Как работает память в Python и сборщик мусора?</summary>

В CPython есть reference counting плюс garbage collector для циклических ссылок. Объект удаляется, когда счётчик ссылок падает до нуля, а циклы добирает GC. Практический caveat: финальный `__del__` и циклы ссылок могут делать момент освобождения неочевидным.

</details>

<details>
<summary>Когда и как объекты удаляются сборщиком мусора?</summary>

Сразу при `refcount == 0`, либо позже проходом cyclic GC, если есть циклические ссылки.

На интервью полезно добавить: объекты с циклическими ссылками и `__del__` исторически tricky для GC, поэтому лучше не строить архитектуру на неявном моменте удаления.

</details>

<details>
<summary>Как происходит управление памятью при использовании multiprocessing?</summary>

У каждого процесса своя память. На Unix возможен copy-on-write после `fork`, но как только страницы памяти модифицируются, они начинают дублироваться. Поэтому multiprocessing хорошо подходит для изоляции CPU-задач, но плохо для частого обмена большими объектами.

</details>

<summary>Продвинутое использование Python</summary>

Это скорее маркер следующего блока вопросов, а не самостоятельный теоретический вопрос. Обычно после него спрашивают про межпроцессное взаимодействие, работу с API, type hints и небольшие алгоритмические задачи.

</details>

<details>
<summary>Как организовать передачу информации между двумя приложениями на Python?</summary>

Сначала надо выбрать модель взаимодействия, а уже потом технологию:

- `request/response` -> HTTP или gRPC;
- `async fire-and-forget` -> очередь сообщений или broker;
- `shared state / cache` -> Redis;
- `durable shared storage` -> БД;
- `локальный обмен между процессами` -> socket, pipe, shared memory.

Критерии выбора такие:

- нужна ли синхронность или можно работать асинхронно;
- важнее latency или durability;
- нужны ли гарантии доставки и порядок сообщений;
- какой объём данных передаём;
- нужна ли строгая схема и версионирование контракта;
- это локальная интеграция или сетевой сервис.

Хороший собеседовательный ответ: если нужен быстрый RPC с контрактом — `gRPC`; если нужен слабосвязанный обмен и ретраи — broker; если нужен просто общий источник истины — БД; если это простой локальный сценарий — pipe/socket/file может быть достаточно.

</details>

<details>
<summary>Как скачать и обработать датасет изображений через API сервиса?</summary>

Я бы строил end-to-end pipeline примерно так:

1. пройтись по API c авторизацией, пагинацией и батчевой выгрузкой ID/URL;
2. положить задачи скачивания в очередь;
3. ограничить concurrency и соблюсти rate limit сервиса;
4. для сетевого I/O использовать async или thread pool;
5. сохранять файлы в предсказуемую структуру каталогов и отдельно писать метаданные;
6. валидировать ответ: статус, MIME type, размер, checksum, возможность открыть изображение;
7. CPU-предобработку, например resize или конвертацию, вынести в отдельный этап или process pool;
8. вести checkpointing, чтобы после сбоя продолжить с последнего обработанного батча;
9. логировать permanent failures отдельно от transient failures;
10. собрать мониторинг: throughput, error rate, retry rate, latency, corrupted files.

Ключевые инженерные детали, которые хорошо звучат на собесе:

- `retry` с exponential backoff только для временных ошибок;
- идемпотентная запись, чтобы перезапуск не портил результат;
- deduplication, если один и тот же объект может прийти несколько раз;
- разделение I/O и CPU этапов;
- хранение сырого датасета и обработанной версии раздельно.

</details>

<details>
<summary>Что такое Type Hints и как они используются?</summary>

Это аннотации типов для читаемости, IDE, статического анализа и документации. На рантайме Python не становится строго типизированным автоматически. На собесе полезно сказать, что type hints особенно ценны в больших кодовых базах и для контрактов между модулями, но не заменяют тесты.

</details>

<details>
<summary>Напишите функцию для умножения матрицы на вектор.</summary>

```python
def matvec(matrix, vector):
    return [sum(a * b for a, b in zip(row, vector)) for row in matrix]
```

Сложность: \(O(nm)\), где \(n\) — число строк, \(m\) — длина вектора.

</details>

<details>
<summary>Объедините два отсортированных списка в один без использования внешних библиотек.</summary>

```python
def merge_sorted(a, b):
    i = j = 0
    out = []
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            out.append(a[i])
            i += 1
        else:
            out.append(b[j])
            j += 1
    out.extend(a[i:])
    out.extend(b[j:])
    return out
```

Сложность: \(O(n + m)\) по времени и \(O(n + m)\) по памяти на результат.

</details>

<details>
<summary>Напишите функцию для проверки, является ли массив арифметической прогрессией (массив не отсортирован).</summary>

Идея:

1. найти `min`, `max`;
2. проверить, что \((max-min)\) делится на \(n-1\);
3. вычислить шаг;
4. проверить, что все ожидаемые элементы есть.

Это делается за \(O(n)\) по времени и обычно за \(O(n)\) по памяти через `set`, если не оптимизируем память любой ценой.

</details>

<details>
<summary>Как отсортировать файл размером 100 GB при ограничении RAM 120 GB?</summary>

Если RAM действительно 120 GB, файл можно уместить целиком. Но на практике важно оставить запас под ОС, буферы и сам процесс. Если бы файл был больше доступной памяти, использовали бы external sort:

1. сортировать чанки;
2. писать временные файлы;
3. делать k-way merge.

</details>

<details>
<summary>Как найти медиану, если в SQL или Python нет встроенной функции для этого?</summary>

Нужно либо отсортировать и взять середину, либо использовать `quickselect` для \(O(n)\) среднего времени, либо two-heaps для online-случая. Для одного оффлайн расчёта сортировка обычно достаточно понятна и надёжна.

<details>
<summary>Реализуйте функцию подсчёта recall для двух NumPy-массивов.</summary>

```python
import numpy as np

def recall_score(y_true: np.ndarray, y_pred: np.ndarray) -> float:
    tp = np.sum((y_true == 1) & (y_pred == 1))
    fn = np.sum((y_true == 1) & (y_pred == 0))
    return tp / (tp + fn) if (tp + fn) else 0.0
```

Сложность: \(O(n)\) по времени и \(O(1)\) по дополнительной памяти, если не считать входные массивы.

</details>

<details>
<summary>Что нужно, чтобы отсортировать список элементов в Python? Какова сложность встроенной сортировки?</summary>

Нужно, чтобы элементы были сравнимы или был задан `key`. Встроенная сортировка — Timsort: стабильная, гибридная, очень хорошо работает на частично отсортированных данных. Сложность в худшем и среднем случае \(O(n \log n)\), а на почти отсортированных данных может работать близко к \(O(n)\).

</details>

<summary>Напишите функцию, определяющую максимальную высоту, достигнутую велосипедом (задача с высотой).</summary>

Если есть массив изменений высоты, надо идти префиксной суммой и брать максимум. Сложность: \(O(n)\) по времени и \(O(1)\) по памяти.

</details>

<summary>Задача: найдите максимальную длину монотонного подмассива и верните его индексы.</summary>

Идея — один проход и два состояния: длина текущего неубывающего и невозрастающего отрезка, плюс границы лучшего ответа. Если нужна строгая монотонность, надо отдельно оговорить обработку равных элементов. Сложность: \(O(n)\) по времени и \(O(1)\) по памяти.
